JBoss EAP Documentation Contributing Guide

1. Overview

This document describes the tasks needed to contribute to JBoss EAP documentation.

  • Before you begin, you need to perform the following one time process to install and configure Git.

  • Once you have configured Git and set up your repositories, use this documentation update process to make changes to the JBoss EAP documentation.

  • After you have completed the previous tasks, you should to configure your environment to build the documentation. This allows you to verify the books build successfully and your changes render as expected before you submit your merge request.

  • You must follow our documented merge request workflow when you submit your merge request.

  • This document also contains advanced Git operations such as how to delete branches, how to work with unmerged commits, and how to synchronize your origin master with upstream. It also contains a section on troubleshooting tips.

For more details and background about the GitLab repository structure, see the EAP Docs Writing Workflow document.

Important
 If at ANY TIME you feel lost or don’t know what to do, contact one of the admins (Zach, Sande, David, Angie) or start asking questions on the #eap-docs IRC channel.

2. Install and Configure Git

2.1. Install Git

The Git client allows you to access the eap-documentation GitLab repository from a command line in your local file system.

  1. Set up Git to allow command line access to the repositories.

  2. Set up authentication to GitLab.

2.2. Fork the Upstream Repository

Fork the eap-documentation upstream repository to create a copy under your own GitLab ID. This allows you to work on multiple features and push changes to branches in your own GitLab instance so you don’t have to worry about losing work. When you are ready, you can request the changes to be merged back into the upstream repository.

  1. Open a browser and navigate to the upstream repository located at https://gitlab.cee.redhat.com/red-hat-jboss-enterprise-application-platform-documentation/eap-documentation.

  2. Click the Fork link located to the left of the SSH link on the same line as the git@gitlab.cee.redhat.com:red-hat-jboss-enterprise-application-platform-documentation/eap-documentation.git URL.

    Figure 1. Fork Repository Link

  3. Choose your user account for the location of the forked repository. This creates your own copy of the repository under your own GitLab ID.

2.3. Clone Your Forked Repository

Clone your forked repository to bring your GitLab repository files to your local machine. Your forked repository is now the origin repository for your local files.

  1. Open a terminal and navigate to the directory where you want to create the new repository folder.

  2. Type the following command.

    $ git clone git@gitlab.cee.redhat.com:USERNAME/eap-documentation.git

  3. Navigate to the newly created eap-documentation folder.

    $ cd eap-documentation/

For more information about forking and cloning, consult the official Gitlab documentation.

2.4. Add the Upstream as a Remote Repository

Once you have your fork checked out and cloned locally, add the upstream repository as a remote:

  1. List the current list of remote repositories:

    $ git remote -v
origin git@gitlab.cee.redhat.com:zrhoads/eap-documentation.git (fetch)
origin git@gitlab.cee.redhat.com:zrhoads/eap-documentation.git (push)

  2. Add the upstream as a remote repository and fetch its contents. This allows you to check out and work with the latest source code.

    $ git remote add -f upstream  git@gitlab.cee.redhat.com:red-hat-jboss-enterprise-application-platform-documentation/eap-documentation.git

  3. Verify the new remote was added:

    $ git remote -v
origin git@gitlab.cee.redhat.com:zrhoads/eap-documentation.git (fetch)
origin git@gitlab.cee.redhat.com:zrhoads/eap-documentation.git (push)
upstream git@gitlab.cee.redhat.com:red-hat-jboss-enterprise-application-platform-documentation/eap-documentation.git (fetch)
upstream git@gitlab.cee.redhat.com:red-hat-jboss-enterprise-application-platform-documentation/eap-documentation.git (push)

  4. If the upstream repository is moved, you can change the upstream URL using the following command.

    $ git remote set-url upstream git@gitlab.cee.redhat.com:red-hat-jboss-enterprise-application-platform-documentation/eap-documentation.git

  5. Use the following command any time you need to fetch the latest source code locally.

    $ git fetch upstream

3. Documentation Update Process

You should now have a clone of your forked repository on your local machine and added the upstream repository as a remote. Let’s get to work on committing and bubbling those changes upstream! The basic process is as follows:

  • Check out a topic branch from the upstream repository

  • Make your changes and add them to Git

  • Commit your changes and push them to your own repository

  • Verify your changes and open a merge request to upstream

  • Fix issues noted in your merge request

You may want to need to do some more advanced Git operations, such as delete branches, access unmerged commits, or sync up your GitLab repository master branch with the official upstream repository. There is also a list of useful Git commands. Feel free to add to this list.

The following diagram visually displays the GitLab workflow used to update JBoss EAP documentation. Use it as a reference as you go through the process in the next section.

3.1. Check Out a Topic Branch from the Upstream Repository

  1. Fetch the latest from the upstream repository to make sure you are working with the latest code base.

    $ git fetch upstream

  2. Create a local topic branch based on an upstream branch. The topic branch name should be descriptive, for example, the JIRA, bugzilla, or feature name.

    $ git checkout -b TOPIC_BRANCH_NAME upstream/master
    Note
    		 You can check out multiple topic branches and work on multiple features at one time, with no impact on each other.

3.2. Make Your Changes and Add Them to Git

  1. When you have completed your changes, build the book locally to make sure there are no errors or warnings and that the HTML renders successfully. Since a topic may be used in multiple books, be sure to build all of the books if you change a shared topic!

  2. When you are sure the book builds and renders successfully, check the status of your topic branch. The status command tells you the topic branch you are on and lets you know which files have been changed or removed. It also lists any new untracked files you may want to add.

    $ git status

    You should see something like the following:

    Changes not staged for commit:
  (use "git add/rm <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

	modified:   internal-resources/contributor-guide.adoc

Untracked files:
  (use "git add <file>..." to include in what will be committed)

	internal-resources/some-new-test-guide.adoc

  3. Follow the instructions to add changes, add files, or unstage changes.

    $ git add internal-resources/contributor-guide.adoc
$ git add internal-resources/some-new-test-guide.adoc

3.3. Commit Your Changes and Push Them to Your Own Repository

  1. Commit your changes to your local file system.

    $ git commit -m "commit message"

  2. Use Git status to verify all your changes are commited.

    $ git status

    You should see this result:

    nothing to commit, working directory clean

  3. Push your changes to your forked repository (origin).

    $ git push origin HEAD
    Note
    		 This is the equivalent the command git push origin TOPIC_BRANCH_NAME. HEAD points to the top of the current branch so you do not need to remember or type the topic branch name.

3.4. Verify Your Changes and Open a Merge Request to Upstream

After you push your changes to your forked repository (origin), you should verify the changes are correct and then request they be merged back into the upstream repository.

Using the GitLab Web interface:

  1. Navigate to your forked repository, for example, https://gitlab.cee.redhat.com/zrhoads/eap-documentation.

  2. Click Merge Requests on the left side

  3. Click the New Merge Request button on the right side

  4. Select the source and destination branches:

    1. USERNAME/eap-documentation and your branch for source (e.g. zrhoads/eap-documentation and myBugFix)

    2. red-hat-jboss-enterprise-application-platform-documentation/eap-documentation and the corresponding upstream branch for destination (e.g. red-hat-jboss-enterprise-application-platform-documentation/eap-documentation and master)

  5. Click Compare Branches

  6. Click on the Changes tab to view the changes that were committed. Make sure only the changes you expect are in the commit. If you see a modified file in the commit that does NOT belong, see Remove a File From a Commit for instructions to remove it.

  7. Fill in the details about your merge request, assign it to an administrator, and click Submit new merge request.

For more details on creating a merge request, see the GitLab documentation.

3.5. Modify a Merge Request

The administrator or others may make comments and request that you make fixes to your merge request before it is merged. Use the following procedure to make your updates for an existing merge request.

  1. If you have already started making changes in a new branch, you must stash or commit your changes before you continue.

    To stash your changes made in the NEW_BRANCH_NAME branch, type:

    $ git stash save

    Make sure the changes are stashed and nothing shows as modified.

    $ git status

    Later, when you are done with the merged branch and want to return to other branch, recover your changes using the following command.

    $ git checkout NEW_BRANCH_NAME
$ git stash pop

  2. Check out the branch that contains your changes and from which your issued the merge request.

    $ git checkout MERGE_BRANCH_NAME

  3. Make the changes as noted in the merge comments. Follow the instructions above to add the files: Make Your Changes and Add Them to Git

  4. Commit your changes.

    $ git push <-f> origin HEAD
    Note
    		 If you squash commits, you must use the -f flag to force the push to your repository.

  5. The merge request should now reflect your latest updates. You do NOT need to issue another merge request when you use the same branch.

4. Configure Your Environment to Build the Documentation

When you make changes to the documentation, it is a good practice to do a local test build to verify the book builds successfully and renders as you expect before you submit the merge request back to upstream master. This section describes that process.

You can build the books using either AsciiDoctor or ccutil, however, be aware that only ccutil catches build errors like invalid xfrefs and missing includes.

Follow these instructions to install ccutil: https://gitlab.cee.redhat.com/ccs-tools-documentation/docs2drupal-user-guide/blob/master/ccutil-install.adoc

Note
 The ccutil utility requires Ruby. If you have not installed Ruby, you can use this command to install it: dnf -y install ruby

5. Build the Documentation

Build All Books Using the Script

To build all of the books, navigate to the root folder of the GitLab eap-documentation folder and type:

$ scripts/buildGuides.sh

The script builds 2 versions for each guide:

  • An HTML version using AsciiDoc. This provides a build for people who have not installed ccutil.

  • An epub/PDF/HTML version using ccutil

Build a Single Book

You can build a single book using any of the following methods.

  • To build a single book using the build script located in the book folder, navigate to the book folder and execute the buildGuide.sh script.

    $ cd docs/development-guide
$ ./buildGuide.sh

  • You can also execute the same script used to build all books, but pass in the book folder name on the command line.

    $ scripts/buildGuides.sh development-guide

  • To build a book manually using the ccutil utility, navigate to the book folder and type the following command.

    $ ccutil compile --lang en_US --main-file GUIDE_NAME.adoc

  • To build a book manually using AsciiDoctor, navigate to the book folder and type the following command.:

    $ asciidoctor -t -dbook -a toc -o ../html/GUIDE_NAME.html GUIDE_NAME.adoc
Note

You may see the following warnings and errors. They are known warnings and you can ignore them.

WARNING: Image missing: tmp/en-US/xml/images/IMAGE_NAME.png
RelaxNG Validation failed for '/home/username/path-to-doc/build/tmp/en-US/xml/master.xml':
IGNORING: Validation is broken, see BZ #1097495

6. Merge Request Workflow

This section describes the workflow you should follow when you submit a merge request to the upstream master branch.

  1. Upon submitting a merge request, add the Needs Peer Review label and tag someone on the team to review the content.

  2. If the updates are a result of a JIRA, open the JIRA, click Link Pull Request, and paste the link to the GitLab merge request in the text box provided.

  3. The reviewer should review the content and do one of the following.

    • If the content looks good, remove the Needs Peer Review label, add the Ready to Merge label, and comment that the code has been reviewed and looks good to merge.

    • If changes are required, leave the Needs Peer Review label in place, add the Feedback Provided label, and provide good comments about what needs to be changed.

  4. If necessary, the writer receiving feedback should make any necessary changes, remove the Feedback Provided label to notify the reviewer that updates were made, and then tag the reviewer to review any updates.

    Important

    The writer requesting the merge should NOT add the Ready to Merge label at this point. The updated content still needs review and it is up to the reviewer to decide when it is ready.

    This process continues until the reviewer marks the merge request Ready to Merge.

  5. When a merge request is in a Ready to Merge state, a GitLab administrator does final review.

    • If the content looks good, the administrator can merge the request.

    • If the administrator finds a problem, the Feedback Provided label is added and comments are provided about what needs to be changed.

  6. After the request is merged to master, open the related JIRA, click Pull Request Closed or Workflow → Resolve Issue, add a comment and the link where the modified content can be previewed on the QA preview server (https://access.qa.redhat.com/documentation).

7. Advanced Git Operations

7.1. Rename a File or Directory

At some point, you may want or need to rename a file or a directory that is already committed and merged to the upstream repository.

  • To rename a file, use the syntax git mv OLD_FILE_NAME NEW_FILE_NAME.

    $ git mv  migrate-entity-beans.adoc migrate-entity-beans-to-jpa.adoc

    When you do a git status, you should see the following result.

    renamed: migrate-entity-beans.adoc -> migrate-entity-beans-to-jpa.adoc

  • To rename a directory, use the syntax git mv OLD_DIRECTORY_NAME NEW_DIRECTORY_NAME

    $ git mv dev-guide-webservices dev-guide-web-services

    When you do a git status, you should see a rename for every subdirectory and file within the renamed directory.

    renamed:    docs/dev-guide-webservices/buildGuide.sh -> docs/dev-guide-web-services/buildGuide.sh
renamed:    docs/dev-guide-webservices/master-docinfo.xml -> docs/dev-guide-web-services/master-docinfo.xml
renamed:    docs/dev-guide-webservices/master.adoc -> docs/dev-guide-web-services/master.adoc
renamed:    docs/dev-guide-webservices/topics -> docs/dev-guide-web-services/topics
Note
 If you modify a file that is included in a rename command, instead of "renamed:" you may instead see that the original file was deleted and the new file is added. This is okay and is just the way Git reports the change.

7.2. Always Rebase Instead of Pull

Warning

Never do a 'git pull' command!

If at any point Git suggests you do a git pull to bring in upstream changes, DO NOT DO IT! This approach fetches and merges all upstream commits made after your checkout into your local branch and can make the commit history of the branch invalid.

Instead, always use the following commands, which rewind and apply your changes on top of the latest upstream source:

$ git fetch upstream
$ git rebase upstream/master

7.3. Rebase Upstream Changes into Your Branch

If you work in a branch for a while, it is good practice to bring down the latest changes from the upstream repository and test them with your changes before you push to your repository and request a merge. This describes the process to do this.

  1. Add and commit any changes you have made in your branch.

    $ git add file1.adoc
$ git add file2.adoc
$ git commit -m "Your commit message"

  2. Fetch the latest from source from the upstream repository and rebase.

    $ git fetch upstream
$ git rebase upstream/master

  3. If the rebase is successful and there are no conflicts, you see a message similar to the following.

    First, rewinding head to replay your work on top of it...
Applying: "Your commit message"

    If you see "CONFLICT" messages and it fails to merge the changes, you must fix the conflicts and complete the rebase before you continue. See Fix Rebase Merge Conflicts for instructions to resolve the conflicts.

  4. At this point, it is a good idea to display the commit history.

    $ git log

    The commit history should show your commit, followed by a commit history that matches the commit history in the upstream repository.

  5. Push the rebased branch to your origin HEAD

    $ git push origin HEAD

    If you pushed this branch to origin before you rebased, you must use the -f argument to force the new rebased changes into your branch:

    $ git push -f origin HEAD

    Alternatively, you can push it to a branch with a different name, and then close the original merge request and issue a new request for the new branch name.

    $ git push origin HEAD:branch-name-rebased

  6. Using the GitLab Web interface, navigate to your forked repository, click Create Merge Request for this update, verify the changes look correct, and submit the new merge request.

7.4. Fix Rebase Merge Conflicts

Warning

If at any point Git suggests you do a git pull to bring in upstream changes, DO NOT DO IT! See Always Rebase Instead of Pull for details.

When you rebase your changes to the upstream/master branch, you may see error messages like the following:

First, rewinding head to replay your work on top of it...
Applying: Testing a change that should cause a conflict
Using index info to reconstruct a base tree...
M	docs/login-module-reference/master.adoc
Falling back to patching base and 3-way merge...
Auto-merging docs/login-module-reference/master.adoc
CONFLICT (content): Merge conflict in docs/login-module-reference/master.adoc
Failed to merge in the changes.
Patch failed at 0001 Testing a change that should cause a conflict
The copy of the patch that failed is found in:
   /home/sgilda/GitRepos/eap-docs-sandbox/.git/rebase-apply/patch

When you have resolved this problem, run "git rebase --continue".
If you prefer to skip this patch, run "git rebase --skip" instead.
To check out the original branch and stop rebasing, run "git rebase --abort".

Files with conflicts are noted with the message "CONFLICT (content): Merge conflict in". In the example above, you see the message CONFLICT (content): Merge conflict in docs/login-module-reference/master.adoc.

You must manually edit and fix any conflicted files before you can continue.

  1. Open any conflicted files in an editor.

    1. Any changes made in the upstream version of the file are preceded by <<<<<<< HEAD and followed by =======.

    2. Any changes made to the file in your commit follow the ======= and end with >>>>>>> COMMIT_MESSAGE.

  2. Assume, in this example, you find the following conflict in the file.

    End of previous table

<<<<<<< HEAD
=== PropertiesRolesMapingProvider
=======
==== PropertiesRolesMappingProvider
>>>>>>> Testing a change that should cause a conflict
*Code*: PropertiesRoles

    • The change in the upstream/master branch was to adjust a heading up a level from ==== to ===.

    • The change in the current commit was to fix a typo by changing ProperitesRolesMapingProvider to PropertiesRolesMappingProvider.

      In this case, neither line is correct. The correct fix for this conflict is to keep the level at === but fix the typo.

      === PropertiesRolesMappingProvider

  3. Make the changes to the correct lines in the file. Be sure to remove the "<<<<<<< HEAD", "=======", and ">>>>>>> Commit message" delimiting lines and save the file.

  4. Do a git status. You will see something like the following.

    $ git status
rebase in progress; onto f80b096
You are currently rebasing branch 'test-merge-conflict' on 'f80b096'.
  (fix conflicts and then run "git rebase --continue")
  (use "git rebase --skip" to skip this patch)
  (use "git rebase --abort" to check out the original branch)

Unmerged paths:
  (use "git reset HEAD <file>..." to unstage)
  (use "git add <file>..." to mark resolution)

	both modified:   docs/login-module-reference/master.adoc

no changes added to commit (use "git add" and/or "git commit -a")

  5. The conflicted file appears under "Unmerged paths:" with a directive to use "git add <file>" to mark resolution.

    $ git add docs/login-module-reference/master.adoc

    The git status command will now show the file modified and staged for commit.

  6. Now that the conflict is resolved you can tell Git to continue processing the rest of the rebase.

    $ git rebase --continue

  7. At this point, if you do a git log, the commit history should show your commit, followed by a commit history that matches the commit history in the upstream repository.

If you started the rebase process, but do not wish to resolve the merge conflicts at this time, you can abort the rebase with the following command.

$ git rebase --abort

Squash Multiple Commits into One

If you have made multiple commits by the time you have completed a your changes, it is a good idea to first rebase the latest upstream changes into your branch and then "squash" your commits into one commit before you request a merge to the upstream repository.

There are 2 ways to squash commits.

  • You can use the "reset --soft" command.

  • Use the "rebase -i" command.

7.5. Remove a File From a Commit

When you review your merge request, you may find you have mistakenly added a file to the commit that should not be there. You can remove it using the following commands.

$ git reset --soft HEAD^
$ git reset HEAD path/to/file/to/remove
$ git commit -c ORIG_HEAD

If you have already pushed your changes to your repository, you must use the -f option when you push your changes again.

$ git push -f origin HEAD

If you are not sure your new commit is correct and want to preserve the original branch, you can push the changes to a new branch instead.

$ git push origin HEAD:new-branch-name

For more information, see: http://stackoverflow.com/questions/12481639/remove-files-from-git-commit

7.6. Create a Branch That Accesses Unmerged Commits

Because we use the topic-based approach, in most cases we can work autonomously and not need changes made by others. But in some situations you may need changes you or a coworker has committed but not yet merged. One example of this is if another write creates and adds a lot of new topics to the docs/topics directory and issues a merge request. If you need to make update the master.adoc with those topics, you need access to those new topics to test how they render in the build.

7.6.1. Access Your Own Unmerged Commits

This is the process you should use if you need commits you have submitted in a merge request that is not yet merged.

  1. Check out a new topic branch from upstream/master as you normally do.

    $ git fetch upstream
$ git checkout -b NEW_TOPIC_BRANCH upstream/master

  2. Rebase to bring in the changes that are in your outstanding origin/MERGE_REQUEST_BRANCH branch.

    $ git rebase origin/MERGE_REQUEST_BRANCH
   (you'll see the following response)
   First, rewinding head to replay your work on top of it...
   Fast-forwarded NEW_TOPIC_BRANCH to origin/MERGE_REQUEST_BRANCH

7.6.2. Access Another Writer’s Unmerged Commits

This is the process you should use if you need commits another writer has submitted in a merge request that is not yet merged.

  1. Check out a new topic branch from upstream/master as you normally do.

    $ git fetch upstream
$ git checkout -b NEW_TOPIC_BRANCH upstream/master

  2. If you have not yet added that writer’s remote repository, add it now.

    $ git remote add -f zrhoads git@gitlab.cee.redhat.com:zrhoads/eap-documentation.git

  3. Rebase to bring in the changes that are in that user’s outstanding origin/MERGE_REQUEST_BRANCH branch.

    $ git rebase zrhoads/MERGE_REQUEST_BRANCH
   (you'll see the following response)
   First, rewinding head to replay your work on top of it...
   Fast-forwarded NEW_TOPIC_BRANCH to zrhoads/MERGE_REQUEST_BRANCH

7.6.3. Push the New Branch to Origin and Request a Merge.

When you are finished, you can commit and push your changes to origin HEAD. It will show the 2 commits: the one from the rebase and the one from your own changes. You don’t need to worry about that because the administrator will resolve them at merge time.

If the commit is merged by the time you check in your changes and you do want to resolve it before you issue the merge request, follow this process.

  1. Fetch the latest from upstream and rebase the branch to upstream/master.

    $ git fetch upstream
$ git rebase upstream/master
    Warning

    If at any point Git suggests you do a git pull to bring in upstream changes, DO NOT DO IT! See Always Rebase Instead of Pull for details.

  2. You should see the following mesages

    First, rewinding head to replay your work on top of it...
Applying: COMMIT_MESSAGE.

  3. Do a git status to make sure there are no conflicts. If there are conflicts, fix the files with conflicts and then add them using git add.

  4. Push your changes to origin HEAD as you normally do.

7.6.4. Review Another Writer’s Unmerged Commits

This is the process you should use if you need to review commits another writer has submitted in a merge request that is not yet merged.

  1. If you have not yet added that writer’s remote repository, add it now. The user’s GitLab repository URL is displayed at the top of their repository page, for example, https://gitlab.cee.redhat.com/zrhoads/eap-documentation

    $ git remote add -f zrhoads git@gitlab.cee.redhat.com:zrhoads/eap-documentation.git

  2. If you had previously added the user’s remote repository, you must now fetch the latest updates.

    $ git fetch zrhoads

  3. Check out a new topic branch from the user’s branch with the commit you need to review.

    $ git checkout -b REVIEW_BRANCH zrhoads/MERGE_REQUEST_BRANCH

  4. Fetch the latest source from upstream and rebase to bring in those change to be sure nothing has changed that can cause a conflict.

    $ git fetch upstream
$ git rebase upstream/master
   (you'll see the following response)
   First, rewinding head to replay your work on top of it...
   Fast-forwarded REVIEW_BRANCH to upstream/master

  5. Build the guide as you normally do and review the changes.

7.7. Renaming Branches

If you need to work in a topic branch for a while but are not ready to issue a merge request, you may want to name your branch in a way to indicate that it is not ready to be merged. For example:

$ git fetch upstream
$ git checkout -b JBEAP-9999-WIP upstream/master

As you modify content, commit, and push to origin HEAD, reviewers can access your content in the JBEAP-9999-WIP branch of your remote repository. When you are finished and ready to merge, you can rename your local branch using the syntax git branch -m CURRENT_BRANCH_NAME NEW_BRANCH_NAME.

$ git branch -m JBEAP-9999-WIP JBEAP-9999

When you push your changes to origin HEAD, your remote origin repository will now contain both the old branch and the new branch names.

Tip
 Renaming branches is also useful if you want to track large changes in stages or rework the content in a topic but want retain the old versions for later comparison.

7.8. Deleting Branches

Once your merge request has been merged into upstream and you no longer need a topic branch, you can delete the local branch and the corresponding remote origin repository branch. You also want to clean up the stale remote references.

Warning
 Once a branch has been deleted, it cannot be restored. Be certain that you no longer need any changes on the branch before deleting it!

7.8.1. Delete the Branch from Your Remote Origin Repository

  1. Open a browser and access your repository URL, for example:

    https://gitlab.cee.redhat.com/zrhoads/eap-documentation/

  2. The top right side of the page displays links for commits, branches, and tags. Click on the branches link.

  3. You are presented with a page listing the branches in your repository. Find the branch you want to delete and click the red trash can icon to remove it. You are presented with a dialog asking "Removed branch cannot be restored. Are you sure?". Click OK if you are sure you want to delete the branch.

7.8.2. Delete the Branch from Your Local File System

Note
 When you delete the branch from your remote origin repository, it does not delete the branch on your local files system. This is why the following steps are necessary.

  1. Verify that you no longer need any of the changes on this topic branch.

  2. You can not delete the branch you have currently checked out, so you must switch to a topic branch other than the one you are deleting.

    $ git checkout master

  3. Delete the topic branch

    $ git branch -d TOPIC_BRANCH

    You may encounter the following error:

    $ git branch -d TOPIC_BRANCH
error: The branch 'TOPIC_BRANCH' is not fully merged.
If you are sure you want to delete it, run 'git branch -D TOPIC_BRANCH'.

    1. If your merge request to upstream was recently accepted, you may first need to fetch upstream for git to realize the changes were merged.

      $ git fetch upstream
$ git branch -d TOPIC_BRANCH

    2. This can also happen if other changes were committed to the upstream repository after you checked out your branch but before you submitted the merge request. In this case the administrator might have had to rebase your merge request before merging and it doesn’t recognize that it was merged. Verify that your commit is in the upstream repository before you continue with the delete.

      $ git branch -D TOPIC_BRANCH

    3. If you want to discard changes in that branch, you can force the deletion regardless of merge status.

      $ git branch -D TOPIC_BRANCH

  4. List all of your branches to verify that the branch has been deleted.

    $ git branch
  master
  OTHER_BRANCH

7.8.3. Remove Stale References to Remote Branches

When you delete the branches, it does not delete the remote tracking information. You must manually remove any remote-tracking references that no longer exist on the remote repository. It is not necessary to do this every time you delete a branch, but it’s a good practice to do this on a regular basis.

You remove remote tracking references using the git remote prune REMOTE_NAME command.

$ git remote prune origin
  x [deleted]         (none)     -> origin/TOPIC_BRANCH
  Pruning origin
  URL: git@gitlab.cee.redhat.com:zrhoads/eap-documentation.git
   * [pruned] origin/TOPIC_BRANCH
Note

If you regularly check out branches from repositories other than your own, you can use this command to clear obsolete tracking references to them as well.

For example: $ git remote prune zrhoads
Note
 See Cleaning Up Your Local Repository: The Easy Way for a great tip on making this easy, as well as cleaning up local branches that have been merged into master.

7.9. Keeping Origin Master in Sync with Upstream Master

While you should never work in the master branch, you may want to sync the master branch in your forked repository to mirror that of the master branch in the upstream repository. You can do that using either of the following procedures.

7.9.1. Use the Rebase Method

This method rebases any changes upstream into your master.

  1. Check out the branch from origin you want to sync up (e.g. master)

    $ git checkout -b sync-master origin/master

  2. Fetch the latest from upstream

    $ git fetch upstream

  3. Rebase the latest from upstream onto your branch

    $ git rebase upstream/master

  4. Resolve any conflicts. If you have any questions, reach out to one of the admins.

  5. Push the updates to origin (your fork)

    $ git push origin HEAD:master

7.9.2. Use the Reset Hard Method

The method just replaces your master branch with the upstream master.

Warning
 This method overwrites the master branch of your forked repository, so use it only if you do NOT have code you need to preserve.

  1. Check out the master branch

    $ git checkout master

  2. Fetch the latest code from upstream.

    $ git fetch upstream

  3. Reset it to the upstream master

    $ git reset --hard upstream/master

  4. Push the changes back to your forked repository (origin).

    $ git push -f origin HEAD

9. Troubleshooting Tips

  • You see the warning "warning: refname 'upstream/master' is ambiguous." when you attempt to rebase to upstream/master.

    $ git rebase upstream/master
$ git rebase upstream/master
warning: refname 'upstream/master' is ambiguous.
warning: refname 'upstream/master' is ambiguous.
First, rewinding head to replay your work on top of it...

    This usually means you accidently created a branch named upstream/master. This is easy to do if you omit the branch name on the checkout command, for example: git checkout -b upstream/master. To verify this, type:

    $ git branch
branch_01
* current_branch
master
test_branch
upstream/master

    If you see "upstream/master" in the list of branches, simply remove it using:

    $ git branch -d upstream/master

10.1. Display the Git Branch and Modifications

You can enhance Git to make it easier and more useful. This section describes how to add the Git branch name to the command prompt so you are aware of the topic branch you are working with. It also marks the prompt with a trailing asterisk (*) when files have been modified since the checkout command was issued so that is easy to know at a glance that changes were made to the branch.

  1. Download the following script directly from the Git source code repository at https://github.com/git/git/blob/master/contrib/completion/git-completion.bash directly into your home directory.

  2. Copy the following commands into your .bashrc file.

    # Define a variable for the script file
export GIT_COMPLETION_PATH="/home/USER_NAME/git-completion.bash"
# Display the current branch in the prompt and mark with an '*' if files have been modified since the _checkout_ command.
.
if [ -f "$GIT_COMPLETION_PATH" ]; then
  GIT_PS1_SHOWDIRTYSTATE=true
  . "$GIT_COMPLETION_PATH"
  ADD_PS1='$(__git_ps1)'
fi
if [[ ${EUID} == 0 ]] ; then
  PS1="\[\033[01;31m\]\h\[\033[01;34m\] \w\[\033[33m\]$ADD_PS1\[\033[34m\] \$\[\033[00m\] "
else
  PS1="\[\033[01;32m\]\u@\h\[\033[01;34m\] \w\[\033[33m\]$ADD_PS1\[\033[34m\] \$\[\033[00m\] "
fi

When you navigate to your local repository, your prompt should now contain the branch name in parenthesis along with an asterisk if any files have changed.

For a turbo-boosted take on this, you may want to use bash-git-prompt instead.

10.2. Git Bash Aliases (and Tab Completion!)

Originally posted on Lucas' blog.

Also see http://git-scm.com/book/en/v2/Git-Basics-Git-Aliases.

What is a bash alias?

"A Bash alias is essentially nothing more than a keyboard shortcut, an abbreviation, a means of avoiding typing a long command sequence." [src]

To add an alias to your bash profile so you can use it in your terminal, open your ~/.bashrc file and add the alias definitions. Make sure that you open a new terminal after saving changes to your ~/.bashrc file (any terminals already open won’t get the changes).

Some example git aliases are shown below:

# Git
## Git shortcuts
alias g='git'
alias gfu='git fetch upstream'
alias gr='git rebase upstream/master'
alias gpom='git push origin master'
alias gc='git checkout'
alias gs='git status'
alias gd='git diff'
alias gnewbranch='git checkout -b'
alias gnb='git checkout -b'
alias gcommam='git commit -am'
alias gcomma='git commit -a'
alias grmbranch='git branch -D'
alias gbranches='git branch -a'

You can change the aliases to suit your style, or add any others you would like that aren’t included above.

However, one downside of using aliases is that you normally lose the ability to use native tab completion of the commands. For example, you can’t go gc <tab> and get a list of branches in your repo, as you normally can with git checkout <tab>.

To work around this issue, download the git-completion.bash file described in the Git Auto-Completion and Display Git Branch section above.

Then add the following to your ~/.bashrc file:

# Git branch bash completion
if [ -f ~/git-completion.bash ]; then
  . ~/git-completion.bash

  # Add git completion to aliases
  __git_complete g __git_main
  __git_complete gc _git_checkout
  __git_complete gnb _git_checkout
  __git_complete gnewbranch _git_checkout
  __git_complete gm __git_merge
  __git_complete grmbranch _git_branch
fi

Note the paths to the previously saved file, and then also the mapping of the aliases to the auto-complete methods.

For example, you can now type gc <tab> and get a list of branches in the repo.

10.3. Cleaning Up Your Local Repository: The Easy Way

Your local repository can easily get cluttered with a lot of old branches that have already been merged into master, as well remote-tracking branches of origin branches that have been deleted.

Using a bash alias to execute a few commands can make keeping your local repository tidy very easy:

# This deletes local branches that have been merged and deleted from origin
alias gdryclean="git remote prune origin --dry-run; git branch --merged master | grep -v 'master$'"
alias gclean="git remote prune origin; git branch --merged master | grep -v 'master$' | xargs git branch -d"

The first alias doesn’t actually do anything, it is a 'dry run' that shows you what it will do if executed normally (best to be safe!). The second alias is what actually makes the changes.

Each alias executes two commands: the first prunes remote-tracking branches for origin branches that have been deleted; the second deletes any local branches that have been merged into the upstream master.

11.1. Atom

The Atom text edtior was created by github for editing all sorts of source files. Specifically, it supports asciidoc syntax highlighting and html preview. It also has automatically detects if a file is part of a git repo and does highlighting based on its status (very nice for visually seeing what files have changed).

For the most part, Atom can be used out of the box, but there are a few things that should be installed/configured.

11.1.1. Fixing auto whitespace cleanup

Atom will automatically try to "fix" your placement of whitespace. Some people were not happy about this, including your lowly git admins.

To fix this issue:

  1. Navigate to settings

  2. Click on Packages

  3. Search for "Whitespace" and click on the Settings icon of the Whitespace package

  4. Ensure that BOTH Ensure Trailing New Line and Remove Trailing Whitespace are UNCHECKED

11.1.2. Installing asciidoc syntax highlighting

A package has been created for doing sytax highlight of asciidoc files.

To install this package:

  1. Navigate to settings

  2. Click on Install

  3. Search for "language-asciidoc" and click on the Install icon of the language-asciidoc package

11.1.3. Installing asciidoctor preview

To install this package:

  1. Navigate to settings

  2. Click on Install

  3. Search for "asciidoctor-preview" and click on the Install icon of the asciidoctor-preview package

Important
 asciidoctor-preview is not the same as asciidoc-preview. We use asciidoctor to generate our docs, so you need to use asciidoctor-preview.
